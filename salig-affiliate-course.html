Meet the Data Model

The "cast of characters" — the main things the system keeps track of.

The Big Picture

Every application has "entities" — the nouns. In Salig-Affiliate, here's how everything relates:

🏢 Tenant (Your Company)
The top-level organization. Each tenant has its own team, affiliates, campaigns, and data. When you sign up, you create a tenant.
👥 Users (Your Team)
People inside your company who manage the affiliate program — owners, admins, managers. They have different permission levels.
🤝 Affiliates (External Partners)
The people promoting your product. They sign up, get unique links, and earn commissions.
📢 Campaigns (The Rules)
A campaign defines commission rules — rate, type (percentage/fixed), cookie duration, auto-approval settings.
🔗 Referral Links
Unique URLs tied to an affiliate (and optionally a campaign). Contains a special code like ?ref=jamie123

The Tracking Entities

These track the flow from click to money:

👆
Click
Recorded when someone clicks an affiliate's link. Stores IP, user agent, referrer, and timestamp.
🛒
Conversion
A purchase or signup that happened. Links back to the original click (if any) for attribution.
💵
Commission
Money owed to an affiliate. Goes through a lifecycle: Pending → Approved → Paid (or Declined/Reversed).
💳
Payout
When money actually moves to the affiliate. Commissions are grouped into payout batches.

How Click Tracking Works

Following the journey from a click to a recorded event.

The Click Flow

When someone clicks an affiliate link, here's what happens:

1. Link Contains the Code
URL looks like yoursite.com?ref=jamie123. The ref parameter identifies which affiliate gets credit.
2. Server Records the Click
The tracking code (in clicks.ts) receives the referral code, looks up the affiliate, and stores: IP address, user agent (browser info), referrer (where they came from), and timestamp.
3. Deduplication Happens
If the same person clicks multiple times within an hour, only the first click counts. This prevents affiliates from clicking their own links to inflate commissions.
4. Cookie is Set
A browser cookie stores the affiliate ID for the configured duration (usually 30-90 days). If the person comes back later and buys, the affiliate still gets credit.

Real Code: Recording a Click

Here's the actual code that handles click tracking:

CODE 
export const trackClickInternal = internalMutation({
  args: {
    tenantId: v.id("tenants"),
    referralLinkId: v.id("referralLinks"),
    affiliateId: v.id("affiliates"),
    ipAddress: v.string(),
    dedupeKey: v.string(),
  },
  handler: async (ctx, args) => {
    // Check for existing click
    const existingClick = await ctx.db
      .query("clicks")
      .withIndex("by_dedupe_key", q =>
        q.eq("dedupeKey", args.dedupeKey))
      .first();
    
    if (existingClick) {
      return { clickId: existingClick._id, isNew: false };
    }
    
    // Create new click record
    const clickId = await ctx.db.insert("clicks", {
      ...args
    });
    return { clickId, isNew: true };
  },
});
PLAIN ENGLISH

internalMutation: A function that can modify the database. "Internal" means it can only be called by other server functions, not directly from the browser.

args: What information must be provided — tenant, which link was clicked, which affiliate, IP address, and a dedupe key.

withIndex("by_dedupe_key"): Looking up the click using a pre-built index — super fast even with millions of clicks.

.first(): Get the single matching result (or null if none exist).

isNew: false: If a click already exists with this dedupe key, return the existing one — prevents duplicates!

ctx.db.insert: Actually create the new record in the database.

Why Deduplication Matters

⚠️ Without Deduplication
An affiliate could click their own link 100 times from the same IP, and the system would record 100 "clicks." When they refer a real customer, there's no way to tell which clicks were real.
✅ With Deduplication
Only the first click per IP per hour counts. The affiliate can still legitimately click their link to test it, but they can't inflate numbers. When a real conversion happens, the system knows exactly which click to attribute it to.

The dedupe key is typically a combination of IP address + referral link ID + hour of day. This creates a unique "slot" that can only be filled once.

Real Code: Creating a Commission

Here's the actual code that creates commissions with validation:

CODE 
export const createCommission = mutation({
  args: {
    affiliateId: v.id("affiliates"),
    campaignId: v.id("campaigns"),
    saleAmount: v.number(),
  },
  handler: async (ctx, args) => {
    // Get the campaign
    const campaign = await ctx.db.get(args.campaignId);
    
    // Block paused/archived campaigns
    if (campaign.status === "paused") {
      throw new Error("Cannot create...");
    }
    
    // Calculate based on type
    let commissionAmount;
    if (campaign.commissionType === "percentage") {
      commissionAmount = args.saleAmount * (rate / 100);
    } else {
      commissionAmount = rate; // flat amount
    }
    
    // Insert the commission
    return await ctx.db.insert("commissions", {
      ...
    });
  },
});
PLAIN ENGLISH

mutation: A function that changes data — creates, updates, or deletes records.

Input validation: The args section uses v.id() and v.number() to ensure only valid data enters the system.

Fetch the campaign: Look up the campaign to get its commission rules. Can't calculate without knowing the rate!

Status check: If campaign is paused, throw an error. No commissions can be earned on inactive campaigns.

Percentage vs Fixed: For percentage, multiply sale × rate ÷ 100. For fixed, just use the rate directly.

Insert: Finally, create the commission record in the database.

Affiliate Overrides

Sometimes you want to give a specific affiliate a better rate than the campaign default:

💡 Real-World Use Case

Your top-performing affiliate "Jamie" consistently drives $10k/month in sales. You want to reward them with a 25% rate instead of the standard 15%. You create an override on their profile, and now every sale they refers gets the boosted rate automatically.

CODE 
async function getEffectiveCommissionRate(
  ctx, affiliateId, campaignId
) {
  const affiliate = await ctx.db.get(affiliateId);
  const campaign = await ctx.db.get(campaignId);
  
  // Check for override
  const override = affiliate.commissionOverrides
    .find(o => o.campaignId === campaignId
           && o.status === "active");
  
  // Use override if exists, else campaign default
  return override?.rate ?? campaign.commissionRate;
}
PLAIN ENGLISH

Find the affiliate: Look up their full record to find any overrides.

Find the campaign: Get the default commission rate.

Search overrides: Look through the affiliate's override array for one matching this campaign.

The ?? operator: "Use the override rate if it exists, otherwise use the campaign default." This is a handy shorthand.

Fraud Detection

How the system spots and handles cheaters.

The Types of Fraud

Without fraud detection, people can game the system in various ways:

🔄
Self-Referral Fraud
An affiliate buys their own product through their own link. They get a commission they never deserved.
🤖
Bot Traffic
Automated scripts click affiliate links thousands of times to inflate stats and appear more valuable.
🍪
Cookie Stuffing
Secretly dropping affiliate cookies on visitors who never clicked a link, so they get credit for organic sales.
👥
Fake Accounts
Creating multiple fake affiliate accounts to refer each other and pay out commissions to themselves.

How Salig-Affiliate Detects Fraud

🔍
IP Matching
Compares the affiliate's signup IP with buyer IP. Match = suspicious.
📱
Device Fingerprinting
Identifies devices. Same device across many conversions = flag it.
💳
Payment Matching
Compares payout account with buyer payment method. Match = fraud.
⏰
Timing Analysis
Click → purchase in under 5 seconds = likely fake, not human behavior.
📊
Pattern Detection
Unusual conversion rates, same amounts repeatedly, odd geographic clusters.
🛡️
Signal, Don't Block
Flags suspicious activity for human review rather than auto-rejecting.

Real Code: Fraud Signals

The system stores fraud signals on each affiliate profile:

CODE 
fraudSignals: v.optional(v.array(v.object({
  type: v.string(),         // "selfReferral", "botTraffic"
  severity: v.string(),     // "low", "medium", "high"
  timestamp: v.number(),  // when detected
  details: v.optional(...),  // explanation
  reviewedAt: v.optional(...), // human reviewed?
  reviewNote: v.optional(...), // reviewer's notes
})))
PLAIN ENGLISH

Array of signals: Each affiliate can have multiple fraud signals stored.

Type: What kind of fraud — selfReferral, botTraffic, ipAnomaly, manual_suspension.

Severity: How concerning is this? Low/medium/high helps prioritize review.

Timestamp: When was this detected? Useful for pattern analysis over time.

Reviewed: Has a human looked at this? Prevents signals from being ignored.

Review note: What did the reviewer decide? Creates an audit trail.

🎯 Why Signals, Not Auto-Blocking?
False positives are expensive. A real customer misidentified as fraud means lost revenue and a terrible user experience. Signals let humans make the final call, while the system does the heavy lifting of detection.

Multi-Tenancy

How one app serves many companies at once.

What is Multi-Tenancy?

Instead of running separate servers for each customer, one application serves everyone:

🏢 Single-Tenant
  • YourCompany.com has its own server
  • OtherCompany.com has a different server
  • More expensive to maintain
  • Harder to update everyone at once
  • Better isolation/ security
☁️ Multi-Tenant
  • All companies share one infrastructure
  • Each sees only their own data
  • Cheaper, easier to maintain
  • Updates apply to everyone instantly
  • Must carefully isolate data
💡 Think of it like...
A shopping mall (multi-tenant) vs. separate stores (single-tenant). The mall shares the building, parking, utilities — but each store has its own inventory and staff.

How It Works in Salig-Affiliate

1. Tenant ID on Every Record
Every table has a tenantId field. Every query automatically filters by the current user's tenant.
2. Permission Checks
Before any operation, the system verifies the user has permission to access that tenant's data.
3. Data Isolation
A query for "affiliates" only returns affiliates belonging to the logged-in user's tenant.
4. Shared Resources
All tenants share the same servers, database, and code — but their data is logically separated.

Real Code: Tenant Context

CODE 
// Every query starts with getting tenant context
export const listAffiliates = query({
  args: {},
  handler: async (ctx) => {
    // Get current user and their tenant
    const user = await getAuthenticatedUser(ctx);
    const tenantId = user.tenantId;
    
    // Query with tenant filter
    return ctx.db
      .query("affiliates")
      .withIndex("by_tenant",
        q => q.eq("tenantId", tenantId))
      .collect();
  }
});
PLAIN ENGLISH

getAuthenticatedUser: First, figure out who's making the request.

user.tenantId: Extract which tenant they belong to.

withIndex("by_tenant"): Use the tenant index to filter results.

q.eq("tenantId", tenantId): Only return records where tenantId matches.

The magic: This pattern is repeated in EVERY query and mutation. It's impossible to accidentally return another tenant's data.

The Tracking Script

How the SaaS owner adds click tracking to their website.

What is the Tracking Script?

The tracking script is a small piece of JavaScript code that the SaaS owner copies and pastes into their website. It silently tracks clicks and enables affiliate attribution.

🎯 How It Works

When a visitor clicks an affiliate link, the script records the click in the Salig-Affiliate database. Later, when that visitor makes a purchase, the SaaS company's checkout can "ask" the tracking system: "Who referred this customer?" and the system knows who to credit.

1
SaaS Owner
Copies tracking code from dashboard
→
2
Website
Script runs on every page
→
3
Clicks
Recorded in database
→
4
Purchase
Attribution via cookie

The Tracking Code

Here's what the SaaS owner gets to paste into their website:

CODE 
<!-- Salig-Affiliate Tracking Script -->
<script>
  (function() {
    const TRACKING_KEY = "sk_abc123...";
    const ATTRIBUTION_DAYS = 30;
    
    // Extract ref code from URL
    const params = new URLSearchParams(
      window.location.search);
    const refCode = params.get("ref");
    
    if (refCode) {
      // Set cookie for attribution
      document.cookie = "sa_ref=" + refCode +
        "; max-age=" + (ATTRIBUTION_DAYS * 86400);
    }
  })();
</script>
PLAIN ENGLISH

Tracking Key: A unique identifier for this tenant's tracking data. Generated when the SaaS owner first sets up tracking.

Attribution Days: How long after a click the affiliate still gets credit. Usually 30-90 days.

URLSearchParams: Parses the URL to find the "ref" parameter (e.g., yoursite.com?ref=jamie123).

Document.cookie: Sets a browser cookie with the affiliate's code. The "max-age" makes it expire after the attribution window.

The magic: When someone later buys, the checkout code can read this cookie to know who referred them.

Verifying Installation

How does the system know if the tracking script is actually installed?

1. Ping from Website
The tracking script periodically sends a "ping" to Salig-Affiliate's servers, including the tracking key and the current domain.
2. Database Records Ping
Each ping is stored in the trackingPings table with timestamp, domain, and tracking key.
3. Verification Check
When the SaaS owner clicks "Verify," the system looks for recent pings from their domain.
4. Status Updated
If pings are found, the tenant's trackingVerifiedAt timestamp is set. Dashboard shows "Verified ✓"
✅ Why This Matters
If the tracking script isn't actually running, no clicks will be recorded and affiliates won't get credit. Verification gives SaaS owners confidence their tracking is working.

Affiliate Signup & Login

How affiliates register and access their portal.

The Affiliate Portal

Affiliates have a separate login from the SaaS owner. They access a white-labeled portal to track their earnings, get their referral links, and withdraw commissions.

🔗
Get Links
Generate unique referral links for different campaigns
📊
Track Performance
View clicks, conversions, and earnings in real-time
💰
Request Payouts
Submit payout requests when they have earnings

Signup Flow

When someone wants to become an affiliate:

1. Visit Portal
The potential affiliate visits your customized portal URL (e.g., yourbrand.affiliateportal.com or yourdomain.com/affiliates)
2. Fill Registration Form
They provide: name, email, password, and optionally how they plan to promote (YouTube, blog, social media, etc.)
3. Unique Code Generated
The system generates a unique 8-character code (e.g., "JM8K2NPQ") that identifies this affiliate. It's used in referral links.
4. Status = Pending
The affiliate is created with "pending" status. They cannot log in yet — they must be approved first.
5. Welcome Email Sent
An email goes to the affiliate letting them know they'll hear back in 1-2 business days.

Real Code: Affiliate Registration

Here's how the system creates an affiliate account:

CODE 
// Create affiliate with pending status
const affiliateId = await ctx.db.insert("affiliates", {
  tenantId: tenant._id,
  email: args.email,
  name: args.name,
  uniqueCode: uniqueCode,
  status: "pending",  // Must be approved first!
  passwordHash: passwordHash,
  promotionChannel: args.promotionChannel,
});
PLAIN ENGLISH

ctx.db.insert: Creates a new record in the affiliates table.

tenantId: Links this affiliate to the correct company (tenant).

uniqueCode: A cryptographically secure 8-character code. Used in URLs like yoursite.com?ref=JM8K2NPQ

status: "pending": The affiliate can't log in yet. They need approval first.

passwordHash: The password is never stored in plain text — it's hashed using PBKDF2 with 100,000 iterations.

Login Flow

Once approved, how does an affiliate log in?

1. Enter Credentials
Affiliate enters email and password on the login page.
2. Password Verification
The system hashes the entered password and compares it to the stored hash using PBKDF2.
3. Check Status
Verify the affiliate's status is "active" — not pending, suspended, or rejected.
4. Create Session
A unique session token is generated, stored in the affiliateSessions table, and returned to the browser.
5. Session Expires
Sessions last 7 days. After that, the affiliate must log in again.
🔐 Why Not Use Better Auth?
Better Auth handles the SaaS owner's team logins. But affiliates are external users with simpler needs — just email/password, no OAuth, no 2FA required. A custom lightweight auth is more appropriate.

Scale
  • Up to 500 affiliates

  • 10 team members
  • 100 payouts/month
  • 10,000 API calls/month
💡 Why Tiers Matter
Tier limits prevent abuse and ensure fair pricing. A tenant with 10,000 affiliates needs more resources than one with 50. Tier upgrades generate more revenue.

⚠️ Audit Trail
Every action taken during impersonation is logged. Admins can see what the admin did "impersonating" vs what the actual tenant user did. This is critical for security and accountability.

Use case: A tenant reports "commissions aren't calculating correctly." An admin can impersonate that tenant, reproduce the issue, and see exactly what's happening — without needing the tenant to share their screen.
