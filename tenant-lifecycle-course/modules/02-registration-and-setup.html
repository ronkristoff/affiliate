Step 1: Auth, Step 2: App data

Registration is split into two steps because of an architectural boundary. Better Auth manages credentials (email, password) in its own tables. The app needs additional data (company name, domain) that Better Auth doesn't know about. So the frontend calls them sequentially.

CODE — SignUp.tsx (frontend) 

// Step 1: Create auth user via Better Auth
const { data, error } = await authClient
  .signUp.email({
    email: cleanEmail,
    password: values.password,
    name: cleanName,
  });

// Step 2: Create tenant + user in app tables
await completeSignUpMutation({
  email: cleanEmail,
  name: cleanName,
  companyName: values.companyName,
  domain: cleanedDomain,
  plan: selectedPlan,
});
PLAIN ENGLISH

First, send the email and password to Better Auth...

Better Auth hashes the password and stores it safely...

If this fails (email taken, weak password), stop here...

Only if auth succeeds, create the app-level records...

Send the company name, domain, and plan to Convex...

Convex creates the tenant, the user, and initializes counters...

⚠️
Why not do it in one call?

Better Auth's Convex component has its own database tables with a fixed schema. It only accepts email, password, and name. Fields like companyName and domain would cause a validation error if sent to Better Auth. That's why the second step exists.

The domain gatekeeper

The domain field is critical — it's where the tenant's customers actually buy their product. Referral links point there. The tracking snippet runs there. So the system validates it aggressively, both on the frontend and backend.

CODE — convex/users.ts 

function cleanAndValidateDomain(domain: string) {
  let cleaned = domain.toLowerCase().trim();
  cleaned = cleaned.replace(/^https?:\/\//, "");
  cleaned = cleaned.replace(/^www\./, "");
  cleaned = cleaned.split("/")[0];
  cleaned = cleaned.split(":")[0];
  if (cleaned === "localhost")
    throw new Error("localhost is not allowed");
  const ipRegex = /^(\d{1,3}\.){3}\d{1,3}$/;
  if (ipRegex.test(cleaned))
    throw new Error("IP addresses not allowed");
  return cleaned;
}
PLAIN ENGLISH

Normalize: lowercase, strip spaces...

Remove "https://" if someone pasted a full URL...

Remove "www." prefix...

Remove any path after the domain (e.g. "/about")...

Remove port numbers (e.g. ":3000")...

Reject localhost — this needs to be a real website...

Reject IP addresses — they don't have tracking snippets...

Return the clean domain.

After cleaning, the backend checks two things: is this domain already taken by another tenant? and is the format valid? Both checks use database indexes for instant lookups.

Every tenant gets a slug

The slug is a URL-safe version of the company name. If "TaskFlow Pro" signs up, their slug becomes taskflow-pro. If that's taken, it tries taskflow-pro-1, then taskflow-pro-2, and so on — like getting a username on a social platform.

CODE — convex/users.ts 

async function generateUniqueSlug(ctx, name) {
  const base = name.toLowerCase()
    .replace(/[^a-z0-9]+/g, "-")
    .replace(/^-+|-+$/g, "");
  let slug = base, counter = 1;
  while (true) {
    const existing = await ctx.db
      .query("tenants")
      .withIndex("by_slug", q => q.eq("slug", slug))
      .first();
    if (!existing) return slug;
    slug = `${base}-${counter++}`;
  }
}
PLAIN ENGLISH

Convert "TaskFlow Pro" to lowercase and replace spaces/special chars with hyphens...

Strip any leading or trailing hyphens...

Start a counter at 1 for de-duplication...

Loop: check if this slug is already taken...

Use the "by_slug" database index for instant lookup...

If nobody has it, we're done — return it...

Otherwise, append "-1", "-2", etc. and try again...

💡
Indexes make this fast

Without the by_slug index, the database would have to scan every single tenant to check if "taskflow-pro" is taken. With the index, it's an instant lookup — like checking the back of a book instead of reading every page. This matters when you have thousands of tenants.

One signup, four records

When Alex completes registration, the system creates four things in a single mutation:

1

Tenant Record

The workspace itself — name, slug, domain, plan, trial date, status "active".

2

Audit Log Entry

"tenant_created" — records who created what, when. Required for compliance and debugging.

3

Tenant Stats Seed

A denormalized counter row initialized to zero. Tracks affiliate counts, commission totals, etc. without expensive queries.

4

User Record

Alex as an "owner" tied to this tenant. Links their email to the tenantId so they can log in and see their own data.

After all four records are created, Alex is redirected to /onboarding — the next step in their journey where they'll set up tracking and create their first campaign.
