04

Conversions & Commissions

Turning payments into affiliate earnings

This is where the money happens. When a customer buys something on TaskFlow's website, Stripe (or SaligPay) sends a webhook to Affilio. The commission engine processes it and, if attribution works, creates a commission for the referring affiliate.

The system handles six types of webhook events — each with different logic:

💳

payment.updated

One-time payment confirmed → create conversion + commission

🔄

subscription.created

First subscription payment → conversion + initial commission

🔁

subscription.updated

Renewal, upgrade, or downgrade → new conversion + recurring commission (if enabled)

❌

subscription.cancelled

Customer cancels → no new commissions, preserve existing pending ones

💰

refund.created

Money returned → reverse the commission (set status to "reversed")

How commissions are calculated

When a conversion is attributed to Jamie, the system calculates their commission based on the campaign's rules. Campaigns support two commission types:

CODE — convex/commissions.ts 

// Get the effective rate (check for affiliate override)
const effectiveRate = await getEffectiveCommissionRate(
  ctx, affiliateId, campaignId
);

// Calculate commission amount
let commissionAmount: number;
if (campaign.commissionType === "percentage") {
  commissionAmount = saleAmount * (effectiveRate / 100);
} else {
  commissionAmount = effectiveRate; // flatFee
}
PLAIN ENGLISH

Check if Jamie has a special commission rate for this campaign (otherwise use the campaign default)...

If the campaign is "percentage" type, multiply the sale amount by the rate...

Example: ₱10,000 sale × 10% = ₱1,000 commission...

If the campaign is "flat fee" type, the rate IS the commission...

Example: ₱500 flat fee regardless of sale amount.

💡
Per-affiliate overrides

Alex can give Jamie a special 15% rate even though the campaign default is 10%. The system checks commissionOverrides on the affiliate record first, then falls back to the campaign default. This lets Alex incentivize top performers without changing the campaign for everyone.

⚠️
Auto-approval thresholds

Campaigns can auto-approve small commissions. If autoApproveCommissions is true and the amount is below the approvalThreshold, the commission skips "pending" entirely and goes straight to "approved." Large commissions always require manual review — a fraud safeguard.

Fraud detection: self-referrals

One of the oldest affiliate tricks: the SaaS owner signs up as their own affiliate to get a commission on their own purchases. The system catches this by comparing the customer's data against the affiliate's stored data.

CODE — convex/conversions.ts 

export const detectSelfReferralInternal = internalQuery({
  args: { affiliateId, customerEmail, ipAddress, ... },
  handler: async (ctx, args) => {
    const affiliate = await ctx.db.get(args.affiliateId);
    const reasons: string[] = [];
    if (args.customerEmail === affiliate.email)
      reasons.push("email_match");
    if (args.ipAddress === affiliate.lastLoginIp)
      reasons.push("ip_match");
    return { isSelfReferral: reasons.length > 0, reasons };
  },
});
PLAIN ENGLISH

Define a query that checks if a purchase looks like self-referral...

Takes the affiliate's ID and the customer's details...

Fetch the affiliate's record from the database...

Start with an empty list of suspicious signals...

If the customer's email matches the affiliate's email — red flag!

If the customer's IP matches the affiliate's last login IP — red flag!

Return whether it's self-referral and why.

The stats engine: instant dashboards

Every time a commission is created, approved, reversed, or paid, a corresponding denormalized counter in the tenantStats table is updated in the same transaction. This means the dashboard loads instantly — no expensive COUNT queries.

CODE — convex/tenantStats.ts (hooks) 

export async function onCommissionCreated(
  ctx, tenantId, amount, status
) {
  const stats = await getOrCreateStats(ctx, tenantId);
  const updates: Record<string, number> = {};
  if (status === "pending") {
    updates.commissionsPendingCount = stats.commissionsPendingCount + 1;
    updates.commissionsPendingValue = stats.commissionsPendingValue + amount;
  } else if (status === "approved") {
    updates.commissionsConfirmedThisMonth += 1;
    updates.commissionsConfirmedValueThisMonth += amount;
  }
  await ctx.db.patch(stats._id, updates);
}
PLAIN ENGLISH

When a commission is created, update the stats counters...

Find (or create) the stats row for this tenant...

Prepare an object with only the fields that changed...

If it's a pending commission, increment the pending count and value...

If it's auto-approved, increment the confirmed-this-month counters...

Save all updates in one database operation (same transaction).

💡
Why not just COUNT(*)?

With 50,000 commissions, a COUNT(*) query takes real time. A denormalized counter is a single row lookup — instant. The tradeoff is complexity: every mutation that changes commission status must also update the counter. The AGENTS.md calls this "mandatory mutation hooks" — forgetting one causes dashboard numbers to drift.
