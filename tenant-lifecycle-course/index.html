The Tenant Lifecycle
01

What Is a Tenant?

The starting point of every affiliate empire

Imagine you run a SaaS product called TaskFlow. You want bloggers and influencers to promote your product and earn a cut of every sale they bring in. That's an affiliate program — and Affilio is the software that runs it.

When you sign up for Affilio, the system creates a tenant for you. A tenant is your private workspace — your company's entire affiliate operation lives inside it. Think of it as a self-contained franchise building inside a larger shopping mall.

🏢

Tenant

Your private workspace. Contains your affiliates, campaigns, commissions, payouts, branding — everything.

👥

Users

The people on your team. Owners have full control, managers have partial access. Each user belongs to exactly one tenant.

📢

Campaigns

The deals you offer affiliates. "10% commission on every sale" is a campaign. You can run many simultaneously.

🤝

Affiliates

Your promoters. Each gets a unique referral link and earns commissions when their referrals buy from you.

What lives inside a tenant?

When Alex from TaskFlow signs up, the system creates a single record in the tenants database table. That record holds everything needed to run their affiliate program:

Why "tenant" and not just "user"?

Affilio is a multi-tenant system. One Affilio app serves hundreds of SaaS companies. The "tenant" is the boundary that keeps Alex's data separate from Jamie's. Every query in the system filters by tenant ID first — it's impossible to see another company's data.

The tenant owns everything

A tenant is the parent of nearly every other record in the system. Every index in the schema starts with tenantId, which means every lookup is scoped to one company.

Plans and limits

Every tenant starts on a 14-day free trial with a chosen plan. The plan determines how many affiliates, campaigns, and payouts they can have. These limits are stored in a separate tierConfigs table — not hardcoded.

Step 1: Auth, Step 2: App data

Registration is split into two steps because of an architectural boundary. Better Auth manages credentials (email, password) in its own tables. The app needs additional data (company name, domain) that Better Auth doesn't know about. So the frontend calls them sequentially.

CODE — SignUp.tsx (frontend) 

// Step 1: Create auth user via Better Auth
const { data, error } = await authClient
  .signUp.email({
    email: cleanEmail,
    password: values.password,
    name: cleanName,
  });

// Step 2: Create tenant + user in app tables
await completeSignUpMutation({
  email: cleanEmail,
  name: cleanName,
  companyName: values.companyName,
  domain: cleanedDomain,
  plan: selectedPlan,
});
PLAIN ENGLISH

First, send the email and password to Better Auth...

Better Auth hashes the password and stores it safely...

If this fails (email taken, weak password), stop here...

Only if auth succeeds, create the app-level records...

Send the company name, domain, and plan to Convex...

Convex creates the tenant, the user, and initializes counters...

⚠️
Why not do it in one call?

Better Auth's Convex component has its own database tables with a fixed schema. It only accepts email, password, and name. Fields like companyName and domain would cause a validation error if sent to Better Auth. That's why the second step exists.

The domain gatekeeper

The domain field is critical — it's where the tenant's customers actually buy their product. Referral links point there. The tracking snippet runs there. So the system validates it aggressively, both on the frontend and backend.

CODE — convex/users.ts 

function cleanAndValidateDomain(domain: string) {
  let cleaned = domain.toLowerCase().trim();
  cleaned = cleaned.replace(/^https?:\/\//, "");
  cleaned = cleaned.replace(/^www\./, "");
  cleaned = cleaned.split("/")[0];
  cleaned = cleaned.split(":")[0];
  if (cleaned === "localhost")
    throw new Error("localhost is not allowed");
  const ipRegex = /^(\d{1,3}\.){3}\d{1,3}$/;
  if (ipRegex.test(cleaned))
    throw new Error("IP addresses not allowed");
  return cleaned;
}
PLAIN ENGLISH

Normalize: lowercase, strip spaces...

Remove "https://" if someone pasted a full URL...

Remove "www." prefix...

Remove any path after the domain (e.g. "/about")...

Remove port numbers (e.g. ":3000")...

Reject localhost — this needs to be a real website...

Reject IP addresses — they don't have tracking snippets...

Return the clean domain.

After cleaning, the backend checks two things: is this domain already taken by another tenant? and is the format valid? Both checks use database indexes for instant lookups.

Every tenant gets a slug

The slug is a URL-safe version of the company name. If "TaskFlow Pro" signs up, their slug becomes taskflow-pro. If that's taken, it tries taskflow-pro-1, then taskflow-pro-2, and so on — like getting a username on a social platform.

CODE — convex/users.ts 

async function generateUniqueSlug(ctx, name) {
  const base = name.toLowerCase()
    .replace(/[^a-z0-9]+/g, "-")
    .replace(/^-+|-+$/g, "");
  let slug = base, counter = 1;
  while (true) {
    const existing = await ctx.db
      .query("tenants")
      .withIndex("by_slug", q => q.eq("slug", slug))
      .first();
    if (!existing) return slug;
    slug = `${base}-${counter++}`;
  }
}
PLAIN ENGLISH

Convert "TaskFlow Pro" to lowercase and replace spaces/special chars with hyphens...

Strip any leading or trailing hyphens...

Start a counter at 1 for de-duplication...

Loop: check if this slug is already taken...

Use the "by_slug" database index for instant lookup...

If nobody has it, we're done — return it...

Otherwise, append "-1", "-2", etc. and try again...

💡
Indexes make this fast

Without the by_slug index, the database would have to scan every single tenant to check if "taskflow-pro" is taken. With the index, it's an instant lookup — like checking the back of a book instead of reading every page. This matters when you have thousands of tenants.

One signup, four records

When Alex completes registration, the system creates four things in a single mutation:

1

Tenant Record

The workspace itself — name, slug, domain, plan, trial date, status "active".

2

Audit Log Entry

"tenant_created" — records who created what, when. Required for compliance and debugging.

3

Tenant Stats Seed

A denormalized counter row initialized to zero. Tracks affiliate counts, commission totals, etc. without expensive queries.

4

User Record

Alex as an "owner" tied to this tenant. Links their email to the tenantId so they can log in and see their own data.

After all four records are created, Alex is redirected to /onboarding — the next step in their journey where they'll set up tracking and create their first campaign.

Tracking & Attribution

How the system knows who referred whom

After registration, Alex's first real task is tracking setup. They need to install a small piece of code — the tracking snippet — on their website. Think of it as installing a security camera: it watches every visitor who arrives through an affiliate link.

The system generates a unique public key (sk_ + random hex) that identifies Alex's tenant. This key is embedded in the tracking snippet — it's how the system knows incoming tracking data belongs to TaskFlow, not DesignPro.

CODE — convex/tracking.ts 

function generatePublicKey(): string {
  const array = new Uint8Array(16);
  crypto.getRandomValues(array);
  const hex = Array.from(array)
    .map(b => b.toString(16).padStart(2, '0'))
    .join('');
  return "sk_" + hex;
}
PLAIN ENGLISH

Create a 16-byte random buffer...

Fill it with cryptographically secure random numbers (not predictable)...

Convert each byte to a 2-character hex string...

Join them all together into one string...

Prepend "sk_" (for "secret key") and return it.

Clicks: the breadcrumb trail

Every time someone clicks Jamie's referral link, the system creates a clicks record. This is the breadcrumb trail that connects a future purchase back to Jamie. The record stores:

🔗

referralLinkId

Which specific link was clicked (Jamie could have multiple links for different campaigns)

🤝

affiliateId

Who gets credit — Jamie's affiliate record ID

🌐

ipAddress

The visitor's IP — used for deduplication (don't count the same person clicking twice)

🔑

dedupeKey

A hash of IP + link — prevents click fraud (same person refreshing the page 100 times)

💡
The 30-day cookie window

Clicks expire. If someone clicks Jamie's link today but doesn't buy for 31 days, Jamie doesn't get credit. Each campaign can configure its own cookie duration. The system checks _creationTime on the click record against the current time to enforce this window.

Three ways to attribute a sale

When a payment comes in via webhook, the commission engine tries three methods to figure out which affiliate gets credit — in order of priority:

1

Cookie-based (webhook body)

The tracking snippet embedded a cookie when the visitor clicked Jamie's link. The webhook body includes the affiliate code from that cookie. Fastest, most common path.

2

Coupon code

Jamie shared a discount code "JAMIE15". The customer entered it at checkout. The webhook metadata includes the coupon code. Works even without cookies (social media, email).

3

Email lead matching

Jamie's referral link included Affilio.referral({email}) on the signup form. The customer's email was stored as a "lead." When the webhook arrives with that email, the system matches it back to Jamie.

ℹ️
Coupon beats cookie

If both cookie data AND a coupon code are present, the coupon code wins (ADR-4). This prevents attribution disputes — the explicit coupon choice is considered stronger evidence than a passive cookie.

Organic conversions

If all three attribution methods fail — no cookie, no coupon, no matching lead — the system still records the sale as an "organic conversion." This is important: the SaaS owner still sees their revenue tracked, even if no affiliate gets credit. It helps them understand what percentage of sales come from affiliates vs. natural traffic.

CODE — convex/commissionEngine.ts 

if (!event.attribution?.affiliateCode) {
  console.log("No attribution, logging as organic");
  const conversionId = await ctx.runMutation(
    internal.conversions.createOrganicConversion,
    { tenantId, amount, customerEmail }
  );
  return { conversionId, commissionId: null, organic: true };
}
PLAIN ENGLISH

If we couldn't find any affiliate to credit...

Log it for debugging...

Still create a conversion record (so the dashboard shows the revenue)...

But don't create a commission — no affiliate to pay...

Return with organic: true so the system knows this wasn't affiliate-driven.

04

Conversions & Commissions

Turning payments into affiliate earnings

This is where the money happens. When a customer buys something on TaskFlow's website, Stripe (or SaligPay) sends a webhook to Affilio. The commission engine processes it and, if attribution works, creates a commission for the referring affiliate.

The system handles six types of webhook events — each with different logic:

💳

payment.updated

One-time payment confirmed → create conversion + commission

🔄

subscription.created

First subscription payment → conversion + initial commission

🔁

subscription.updated

Renewal, upgrade, or downgrade → new conversion + recurring commission (if enabled)

❌

subscription.cancelled

Customer cancels → no new commissions, preserve existing pending ones

💰

refund.created

Money returned → reverse the commission (set status to "reversed")

How commissions are calculated

When a conversion is attributed to Jamie, the system calculates their commission based on the campaign's rules. Campaigns support two commission types:

CODE — convex/commissions.ts 

// Get the effective rate (check for affiliate override)
const effectiveRate = await getEffectiveCommissionRate(
  ctx, affiliateId, campaignId
);

// Calculate commission amount
let commissionAmount: number;
if (campaign.commissionType === "percentage") {
  commissionAmount = saleAmount * (effectiveRate / 100);
} else {
  commissionAmount = effectiveRate; // flatFee
}
PLAIN ENGLISH

Check if Jamie has a special commission rate for this campaign (otherwise use the campaign default)...

If the campaign is "percentage" type, multiply the sale amount by the rate...

Example: ₱10,000 sale × 10% = ₱1,000 commission...

If the campaign is "flat fee" type, the rate IS the commission...

Example: ₱500 flat fee regardless of sale amount.

💡
Per-affiliate overrides

Alex can give Jamie a special 15% rate even though the campaign default is 10%. The system checks commissionOverrides on the affiliate record first, then falls back to the campaign default. This lets Alex incentivize top performers without changing the campaign for everyone.

Fraud detection: self-referrals

One of the oldest affiliate tricks: the SaaS owner signs up as their own affiliate to get a commission on their own purchases. The system catches this by comparing the customer's data against the affiliate's stored data.

CODE — convex/conversions.ts 

export const detectSelfReferralInternal = internalQuery({
  args: { affiliateId, customerEmail, ipAddress, ... },
  handler: async (ctx, args) => {
    const affiliate = await ctx.db.get(args.affiliateId);
    const reasons: string[] = [];
    if (args.customerEmail === affiliate.email)
      reasons.push("email_match");
    if (args.ipAddress === affiliate.lastLoginIp)
      reasons.push("ip_match");
    return { isSelfReferral: reasons.length > 0, reasons };
  },
});
PLAIN ENGLISH

Start with an empty list of suspicious signals...

If the customer's email matches the affiliate's email — red flag!

If the customer's IP matches the affiliate's last login IP — red flag!

Return whether it's self-referral and why.

The stats engine: instant dashboards

Every time a commission is created, approved, reversed, or paid, a corresponding denormalized counter in the tenantStats table is updated in the same transaction. This means the dashboard loads instantly — no expensive COUNT queries.

CODE — convex/tenantStats.ts (hooks) 

export async function onCommissionCreated(
  ctx, tenantId, amount, status
) {
  const stats = await getOrCreateStats(ctx, tenantId);
  const updates: Record<string, number> = {};
  if (status === "pending") {
    updates.commissionsPendingCount = stats.commissionsPendingCount + 1;
    updates.commissionsPendingValue = stats.commissionsPendingValue + amount;
  } else if (status === "approved") {
    updates.commissionsConfirmedThisMonth += 1;
    updates.commissionsConfirmedValueThisMonth += amount;
  }
  await ctx.db.patch(stats._id, updates);
}
PLAIN ENGLISH

When a commission is created, update the stats counters...

Find (or create) the stats row for this tenant...

Prepare an object with only the fields that changed...

If it's a pending commission, increment the pending count and value...

If it's auto-approved, increment the confirmed-this-month counters...

Save all updates in one database operation (same transaction).

💡
Why not just COUNT(*)?

With 50,000 commissions, a COUNT(*) query takes real time. A denormalized counter is a single row lookup — instant. The tradeoff is complexity: every mutation that changes commission status must also update the counter. The AGENTS.md calls this "mandatory mutation hooks" — forgetting one causes dashboard numbers to drift.

One batch, many payouts

A batch generation creates three levels of records:

1

PayoutBatch

One record summarizing the entire run: total amount, affiliate count, timestamp. Status starts as "pending." This is the "payroll run" header.

2

Payouts (one per affiliate)

Individual records for each affiliate showing their total. Jamie gets one payout for $24.90. Each links to the batch via batchId.

3

Commission updates

Every commission in the batch gets status: "paid" and batchId set. The totalPaidOut counter in tenantStats is incremented atomically.

💡
The batchId links everything together

The batchId field on commissions and payouts is the thread that ties a specific payroll run together. If Jamie asks "which batch was my $24.90 in?" — the answer is in that field. It's also how the system prevents double-batching: commissions with a batchId are filtered out of future runs.

The engineering behind the curtain

This system makes several deliberate architectural choices that are worth understanding — they come up in almost every production SaaS:

🔒

Tenant Isolation

Every query starts with tenantId. It's impossible for one company to see another's data — not through a bug, not through a hack. This is called multi-tenancy and it's the foundation of every SaaS.

⚡

Denormalized Counters

Stats are pre-computed, not calculated on demand. This trades write complexity (every mutation must update counters) for read speed (dashboards load instantly). The counters are updated in the same transaction as the data change, so they never drift.

🔗

Atomic Mutations

Payout batching either succeeds completely or fails completely. No partial payouts. This uses Convex's transaction guarantee — the database either commits all changes or rolls back to the state before the mutation started.

📝

Audit Everything

Every significant action creates an audit log: tenant created, user removed, commission approved, payout generated, domain changed. This is non-negotiable for a financial system — regulators require it, and it's invaluable for debugging.

🛡️

Layered Fraud Detection

Self-referral checks (email + IP match), click deduplication (same person clicking 100 times), payment status validation (reject "pending" and "failed" payments), and auto-approval thresholds (manual review for large commissions).

🎯

Triple Attribution

Cookie → Coupon → Email lead matching. If the first method fails, try the next. Coupon beats cookie (explicit > implicit). If all fail, record as organic — never lose a sale's data.
