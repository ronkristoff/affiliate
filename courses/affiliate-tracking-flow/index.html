The Tracking Snippet

How a small JavaScript file captures affiliate clicks

What is track.js?

The tracking snippet (called track.js) is a tiny JavaScript program that you paste into your SaaS website's HTML.

Think of it like a security camera for your website — it watches for affiliate activity and reports back to the affiliate system.

🔧
What "snippet" means

In web development, a "snippet" is just a small piece of code that you copy and paste into your site. It's not a full program — it's designed to work alongside your existing website.

Here's the snippet you install on your website:

HTML 
<script src="/track.js" 
        data-key="sk_abc123..." 
        async>
</script>
What it means
script src="/track.js" — Load the tracking code from your affiliate system
data-key="sk_abc123..." — Your unique identifier so the system knows which business this belongs to
async — Load the script in the background so it doesn't slow down your page

You add this once to every page on your website (usually in the <head> section), and it runs automatically for every visitor.

What track.js Actually Does

When track.js loads in a visitor's browser, it does three things:

1
Looks for referral links

Checks if the URL contains a pattern like /ref/AFFILIATE12. If someone clicked an affiliate's link, this pattern appears in the URL.

2
Records the click

If it finds a referral code, it sends a message to your affiliate system saying "Hey, affiliate ABC just sent a visitor!"

3
Saves a cookie

Stores the affiliate code in a small text file in the visitor's browser for 30 days. This way, even if the visitor doesn't sign up immediately, you still know who referred them.

🍪
What's a cookie?

A cookie is a tiny text file that websites store in your browser. It remembers things about you between visits — like what's in your shopping cart, or (in this case) which affiliate sent you.

Inside track.js: The Click Handler

Here's the actual code from track.js that handles referral links:

JavaScript 
// Look for /ref/CODE pattern in the URL
var match = path.match(/\/ref\/([A-HJ-NP-Z2-9]{8})$/i);

// If found, record the click
fetch('/track/click?code=' + code)
What it means
match(/\/ref\/.../) — Looks for URLs ending in /ref/ followed by an 8-character code
fetch('/track/click?code=...') — Sends the affiliate code to the server so it can record the click
⚡
The clean URL trick

After recording the click, track.js uses a clever technique to remove the /ref/CODE from the URL without refreshing the page. The visitor sees a clean URL, but the affiliate still gets credit.

The Group Chat: Click Tracking

Here's how the different parts of the system talk to each other when a click happens:

👤

🖥️

The Commission Tracker

Connecting signup forms to affiliate attribution

The Missing Piece: Signup Detection

Track.js handles clicks — but there's a gap. How does the system know when a visitor actually signs up for your product?

That's where the commission tracker comes in.

🔗
Two halves of tracking

Track.js = "Who sent this visitor?" (captures clicks)
Commission tracker = "Did they actually sign up?" (records conversions)

You add just 3 lines of code to your signup form. When someone completes signup, it fires off a message saying "New customer! Email: [their email]"

The 3-Line Code

Here's the code you add to your signup form:

JavaScript 
if (window.Affilio) {
  Affilio.referral({ 
    email: customerEmail 
  });
}
What it means
if (window.Affilio) — Only run this if track.js is loaded (prevents errors)
Affilio.referral({email: ...}) — Tell the system "A customer just signed up with this email"
customerEmail — The variable containing the customer's email address
📍
Where to put this code

You add this in the same place where your signup form submits — right after the customer's email is collected and before you create their account.

What Happens When It Fires

When Affilio.referral() is called, it does two things:

1
Reads the cookie

Grabs the affiliate code from the cookie that track.js saved earlier. This tells the system which affiliate gets credit.

2
Sends to the server

Sends both the customer's email AND the affiliate code to the server. The server creates a "lead" record linking email → affiliate.

This "lead" record is crucial — it bridges the gap between the click (happened earlier) and the signup (happening now).

The Lead Matching System

The server doesn't just store the lead — it organizes it for later use. Here's what gets saved:

📧
Customer Email

The customer's email address (e.g., john@example.com)

🔗
Affiliate Code

Which affiliate referred this customer (e.g., ALEX1234)

🆔
Click ID

Reference to the original click record

📅
Timestamp

When the signup happened

⚡
The magic: email matching

This lead system has a superpower: even if the customer clears their cookies, when your billing provider sends a payment notification (webhook), the system can match the customer's email to the affiliate who referred them!

Webhooks & Commissions

How billing providers trigger commission creation

The Gap: From Signup to Payment

So far we've tracked:

  • Click → Affiliate code stored in cookie ✓
  • Signup → Email + Affiliate code stored as lead ✓

But here's the thing: signing up isn't the same as paying. A customer might sign up for a free trial, then convert to a paying customer weeks later.

💡
When do commissions actually happen?

Commissions are typically paid when money changes hands — when the customer actually pays for your product. Not when they sign up, but when they pay.

So how does the system know when a payment happens? Webhooks.

What is a Webhook?

A webhook is like an automated phone call between systems. When something happens in one system (like a payment), it automatically calls another system to tell it about it.

📱
Webhook vs API

API = You ask for information (like searching Google)
Webhook = Information comes to you automatically (like a push notification)

In this system:

💳
Stripe or SaligPay

Detects that a customer paid

→
Sends Webhook

HTTP POST to your system with payment details

⚙️
Commission Engine

Processes the payment and creates commission

The Commission Engine

When a payment webhook arrives, the Commission Engine kicks into action.

Here's what it does:

1
Receive payment info

Gets the customer's email and payment amount from the webhook

2
Find the affiliate

Looks up the customer's email in the leads table to find which affiliate referred them

3
Calculate commission

Based on the campaign settings (e.g., 20% commission), calculates how much the affiliate earned

4
Create record

Creates a commission record in the database, waiting for approval

The Group Chat: Webhook to Commission

Here's how the system components talk to each other when a payment triggers a commission:

💳
Stripe

Payment received! Customer john@example.com just paid $99 for a subscription.

⚙️
Commission Engine

Got the webhook. Let me check who referred john@example.com...

🗄️
Database

Lead found! john@example.com was referred by affiliate ALEX1234. Campaign pays 20%.

⚙️
Commission Engine

20% of $99 = $19.80. Creating commission record...

⚙️
Commission Engine

Commission created! Status: Pending (awaiting approval)

The Big Picture

Why disconnecting billing stops commission creation

Why Disconnecting Stripe Stops Commissions

Now you understand the full flow. Let's answer the original question:

⚠️
When you disconnect Stripe

The billing webhook endpoint (`/api/webhooks/stripe`) requires an active Stripe connection to verify and process events. Without it, payment notifications stop arriving.

Here's the chain of events:

1
Stripe is disconnected

You click "Disconnect Stripe" in Settings → Integrations

2
Billing credentials are cleared

The system removes `stripeCredentials`, `stripeAccountId`, and `billingProvider` from your tenant record

3
Webhooks stop working

Stripe can no longer send payment notifications to your endpoint

4
No commission triggers

Without webhook events, the Commission Engine has nothing to process

💡
The key insight

Commissions don't auto-create from thin air — they require billing webhook events. Disconnecting billing = no webhook events = no commissions.

Two Paths to the Same Result

You can use either Stripe or SaligPay as your billing provider. Both work the same way:

💳
Stripe Connect

Connect your existing Stripe account. Payments come from Stripe, webhooks come from Stripe.

💰
SaligPay

Built-in payment processing. All billing managed within the platform.

Either way, you need some billing provider connected for commissions to auto-create.

Stripe Webhook Setup

How Stripe knows where to send payment events

Two Ways to Connect Stripe

There are two methods for connecting Stripe as your billing provider. The difference is in how much manual setup is required.

🚀
OAuth Connect (One-Click)

Click a button and authorize via Stripe. No manual webhook setup needed.

🔧
Manual Webhook Secret

Go to Stripe Dashboard and manually configure the webhook endpoint URL.

💡
Which method is better?

OAuth Connect is easier for users — the platform handles all the configuration. Manual setup is available as a fallback when OAuth isn't configured.

Method 1: OAuth Connect (One-Click)

This is the seamless experience. Here's what happens when you click "Connect Stripe":

1
You click "Connect Stripe"

You're redirected to Stripe's OAuth authorization page

2
Stripe asks for permission

"Allow [Platform Name] to access your Stripe account?"

3
You click "Allow"

Stripe redirects back with an authorization code

4
Server exchanges code for tokens

The platform receives your Stripe account ID and access token

5
Stripe auto-configures webhooks

Stripe knows where to send events for your account — no manual setup!

🔐
What does OAuth authorize?

You're giving the platform permission to receive webhook events from YOUR Stripe account. The platform cannot make charges or access your full Stripe data — only receive payment event notifications.

Method 2: Manual Webhook Secret

If OAuth isn't available, you'll need to configure webhooks manually in the Stripe Dashboard.

Steps:

1
Go to Stripe Dashboard

Navigate to Developers → Webhooks

2
Add an endpoint

Click "Add endpoint" and enter the URL: https://yourapp.com/api/webhooks/stripe

3
Select events

Choose which events to listen for (payment_intent.succeeded, invoice.paid, etc.)

4
Copy the signing secret

After creating the endpoint, copy the "Signing secret" (starts with whsec_)

5
Paste in settings

Enter the signing secret in Settings → Integrations

How Stripe Knows Your Endpoint

The big question: how does Stripe know where to send webhooks?

🔗
OAuth Method

When you authorize via OAuth, Stripe automatically records the webhook endpoint URL. The platform pre-configured this URL when they set up their Stripe Connect application.

🔧
Manual Method

You tell Stripe by entering the endpoint URL in their Dashboard. Stripe stores this URL and sends events there automatically.

In both cases, once configured, Stripe automatically sends events to that endpoint URL whenever payments happen — no ongoing configuration needed.

Events Stripe Sends

When you connect Stripe, it automatically listens for these events related to payments:

payment_intent.succeeded A payment was completed successfully
invoice.paid A subscription invoice was paid (for recurring commissions)
customer.subscription.created New subscription started
customer.subscription.updated Subscription renewed, upgraded, or downgraded
charge.refunded A refund was issued (may affect commissions)
charge.dispute.created A chargeback was initiated

These events are automatically routed to the Commission Engine, which determines whether they should trigger commission creation.

Vanity Coupon Codes

Attribution at checkout time — no cookies needed

The Problem with Cookies

Referral links work great — until they don't. Here's when cookies fail:

1
Customer clears cookies

Visitor clicks your link, but clears browser data before signing up

2
Different device

Visitor clicks on phone, but signs up on desktop

3
Long research phase

Cookie expires (30 days) before customer decides to buy

💡
The solution: Coupon codes

Instead of relying on a cookie stored weeks ago, coupon codes let customers attribute themselves at checkout. The attribution happens at purchase time — not click time.

How Coupon Codes Work

Each affiliate automatically gets a vanity coupon code when they activate.

🏷️
Auto-generated

No setup required! The system creates the code from the affiliate's name + your business name. Example: "Alex Chen" + "MySaaS" → ALEXCHMYSAA

Here's the complete flow:

1
Affiliate shares their coupon code

Listed in their portal or promotional materials

2
Customer enters code at checkout

They type ALEXCHMYSAA in the coupon field

3
Checkout sends code to tracking

Affilio.referral({ email, couponCode: "ALEXCHMYSAA" })

4
System validates and creates commission

No cookie needed — attribution happens at purchase time

Coupon vs Referral Link: Key Differences

🔗
Referral Link

Click-based attribution. Requires tracking snippet + cookies.

🎟️
Coupon Code

Checkout-based attribution. Works without cookies or tracking snippet.

Attribution timing Link = click time | Coupon = purchase time
Requires tracking snippet Link = yes | Coupon = no
Requires cookies Link = yes (30 days) | Coupon = no
Works across devices Link = no | Coupon = yes

👤
Customer

Great deal! I'll use it at checkout.

👤
Customer

Entering coupon code ALEXCHMYSAA at checkout...

🛒
Checkout

Processing payment + calling Affilio.referral({ email, couponCode: "ALEXCHMYSAA" })

⚙️
System

Coupon code validated! ALEXCHMYSAA → Affiliate "Alex Chen". Commission queued!

Priority: Coupon vs Referral Link

What happens if a customer both clicked a referral link AND uses a coupon code?

🏆
Coupon codes WIN

If a coupon code is present, it always takes priority over cookie-based attribution. This is intentional — the coupon shows active purchase intent at checkout.

Attribution priority order:

1
Coupon Code

Highest priority — always wins if present

2
Affiliate Code in Request

Sent directly from tracking code

3
Cookie (from referral link)

Stored from earlier click

4
Email-based Matching

Fallback — looks up email in leads table

5
Organic

No attribution found

Real-World Example

Imagine this scenario:

Scenario

Sarah clicks Alex's referral link and browses your site. She doesn't buy yet. Three weeks later, she sees a discount code from Alex on Twitter. She uses it at checkout.

1
Sarah clicks Alex's link

Cookie stored with Alex's affiliate code. But she doesn't buy.

2
25 days pass (cookie expires in 30)

Cookie still valid but Sarah's moved on.

3
Sarah sees Alex's coupon code on Twitter

Uses it at checkout: ALEXCHMYSAA

4
Alex gets the commission!

Coupon code wins over expired cookie. Alex earns commission even though the cookie was about to expire.

✨
Why this matters

Coupons capture purchase intent at the moment of buying — not just the initial click. This gives affiliates credit when they influence the actual decision, not just the first visit.

Code: How to Pass Coupon Codes

If your checkout form collects a coupon code, pass it to the tracking:

JavaScript 
// At checkout, after collecting coupon code
if (window.Affilio) {
  Affilio.referral({ 
    email: customerEmail,
    couponCode: customerEnteredCoupon  // ← Add this!
  });
}
What it means
email: customerEmail — The customer's email (required)
couponCode: customerEnteredCoupon — The coupon code they entered (optional — system ignores if empty)

The tracking system automatically handles whether to use the coupon code or fall back to cookie attribution.
