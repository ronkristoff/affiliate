Module 1

What Is Affiliate Marketing — And Why Does It Exist?

Before we look at any code, let's understand the problem this entire system was built to solve.

The Problem Every SaaS Owner Faces

Imagine you run a software company — let's call it TechFlow. You've built a great product, but nobody knows about it. You need customers, and the traditional ways to get them are expensive:

📺

Ads

You pay upfront whether anyone buys or not. Risky.

📧

Email Campaigns

People ignore them. Open rates are dropping every year.

🤝

Affiliate Marketing

You only pay when someone actually brings you a paying customer. Zero risk.

💡 Think of it like this

Affiliate marketing is like having a network of freelance salespeople who only get paid when they close a deal. No salary, no benefits — just a commission on each sale they bring in. It's the ultimate performance-based marketing model.

The Core Concept in One Sentence

Affiliate marketing is when a person (the affiliate) promotes a company's product using a special tracking link. When someone clicks that link and buys the product, the affiliate earns a cut of the sale.

1

Affiliate gets a unique tracking link

Like yourproduct.com?ref=jamie123 — this link identifies which affiliate sent the customer.

2

Affiliate shares the link with their audience

On their blog, YouTube channel, social media, or newsletter.

3

A customer clicks and buys

The system records the click, tracks the purchase, and attributes it to the affiliate.

4

Affiliate earns a commission

A percentage or fixed amount of the sale goes into the affiliate's account.

Why Build an Entire Platform for This?

You might wonder: "Can't this just be a spreadsheet?" For 2 affiliates, maybe. But when you have dozens of affiliates, multiple campaigns, recurring subscriptions, fraud detection, and payout batches — you need a real system.

🔗

Tracking

Who clicked what, when, and from where? Every click needs to be recorded and deduplicated.

💰

Commission Calculation

Different campaigns pay differently — percentages, flat fees, recurring rates. The math must be exact.

🛡️

Fraud Prevention

What if an affiliate tries to earn commissions on their own purchases? The system must detect and flag this.

📦

Payout Management

Grouping commissions into batches, tracking which affiliates have been paid, and generating payout reports.

💡
Aha! Moment

This entire codebase — hundreds of files — exists to automate what would otherwise be a nightmare of spreadsheets, manual calculations, and trust issues between SaaS owners and their affiliates.

Meet the Cast: Who's Involved in Affiliate Marketing?

Every affiliate system has four key players. Understanding who they are and what they do is essential to understanding how the code works.

The Four Key Players

🏢

The SaaS Owner ("Alex")

Runs a software company. Wants to grow their customer base without spending money upfront on ads. Creates campaigns, sets commission rates, approves affiliates, and reviews commissions.

📣

The Affiliate ("Jamie")

A content creator, blogger, or marketer. Promotes the SaaS product to their audience. Earns a commission for every customer they bring in. Needs clear dashboards and transparent tracking.

🛒

The Customer

The person who clicks an affiliate's link and buys the product. They may not even know they came through an affiliate link — it's transparent to them.

⚙️

The Platform (This Codebase)

The automated system that connects everyone. Tracks clicks, calculates commissions, detects fraud, processes payouts, and keeps all the data organized and accurate.

One Platform, Many Companies

Here's what makes this system special: it's multi-tenant. That means one installation of this software can serve many different SaaS companies simultaneously, and each company's data is completely isolated from the others.

💡 Think of it like this

Imagine an office building where each company has its own locked floor. They all share the same building (the platform), but TechFlow can't see what's happening on GHL Agency Pro's floor, and vice versa. Every piece of data — every affiliate, every commission — is tagged with a tenantId that keeps it in the right company's "floor."

The Complete Affiliate Journey: From Click to Cash

This is the heart of the system. We'll trace exactly what happens from the moment someone clicks an affiliate link to the moment money lands in the affiliate's account.

The Five-Stage Journey

Every affiliate earning follows the same five-stage pipeline. If any stage breaks, the affiliate doesn't get paid. Let's walk through each one.

1. Click
→
2. Conversion
→
3. Commission
→
4. Approval
→
5. Payout
📋
The chain must stay unbroken

If a click isn't recorded, there's no attribution. If a conversion isn't linked to a click, there's no commission. If a commission isn't approved, there's no payout. Each stage depends on the previous one.

Stage 1: The Click — "Someone Followed the Link"

When a potential customer clicks an affiliate's tracking link, the system records this event. This is the first link in the attribution chain.

💡 Think of it like this

Imagine a retail store where every salesperson has their own colored nametag. When a customer walks in wearing a "referred by Jamie" sticker, the store knows exactly who to credit if that customer buys something. The click is the moment the customer puts on that sticker.

Here's what the system does when a click happens:

1

Resolve the referral link

The system looks up the affiliate code in the URL and finds which affiliate it belongs to. It also checks if the affiliate is still active (suspended affiliates' links return 404).

2

Check for duplicate clicks

If the same person clicks the same link multiple times within a short window, only the first click counts. This prevents inflating click numbers.

3

Record the click

The system stores the click with the affiliate ID, IP address, user agent, referrer, and a unique deduplication key.

4

Set an attribution cookie

A cookie is placed in the customer's browser so that when they eventually buy, the system knows which affiliate to credit. This cookie has an expiration window (typically 30 days, configurable per campaign).

// convex/clicks.ts — trackClickInternal

// Check for existing click by dedupeKey
const existingClick = await ctx.db
  .query("clicks")
  .withIndex("by_dedupe_key",
    (q) => q.eq("dedupeKey", args.dedupeKey)
  )
  .first();

if (existingClick) {
  // Duplicate — skip, don't create new record
  return { clickId: existingClick._id, isNew: false };
}

// Not a duplicate — create new click record
const clickId = await ctx.db.insert("clicks", {
  tenantId: args.tenantId,
  referralLinkId: args.referralLinkId,
  affiliateId: args.affiliateId,
  ipAddress: args.ipAddress,
  dedupeKey: args.dedupeKey,
});
Look up the clicks table using the deduplication key — a unique identifier for this specific click event.
If a click already exists with this key, return the existing record and mark it as not new. This prevents double-counting.
If it's a new click, insert a fresh record into the database with all the tracking data: which tenant, which affiliate, the IP address, and the dedup key.
💡
Why deduplication matters

Without deduplication, a single person refreshing the page 10 times would count as 10 clicks. This would inflate metrics and make it impossible to calculate accurate conversion rates. The dedupeKey is typically a hash of the IP address + timestamp window.

Stage 2: The Conversion — "They Actually Bought Something"

A click means nothing if the person doesn't buy. A conversion is when a click results in a paying customer. This is where the money happens.

Conversions can come from three different sources:

🍪

Cookie Attribution

The customer had an affiliate's tracking cookie in their browser when they purchased. The system reads the cookie and attributes the sale to that affiliate.

🔔

Webhook Attribution

The payment system (SaligPay) sends a webhook notification saying "payment received." The webhook includes the affiliate code in its metadata, so the system knows who to credit.

🌿

Organic (No Affiliate)

A customer bought without coming through any affiliate link. The system still records this as an "organic conversion" for analytics, but no commission is created.

⚠️
The Cookie Window

Affiliate cookies don't last forever. Each campaign has a cookieDuration (default 30 days). If a customer clicks an affiliate link but doesn't buy within 30 days, the cookie expires and the affiliate gets no credit. This is configurable per campaign.

// convex/conversions.ts
// findRecentClickInternal — attribution chain

// Use campaign-specific window or default 30 days
const windowDays = args.attributionWindowDays ?? 30;
const cutoffTime = Date.now() - attributionWindowMs;

// Find recent clicks for this affiliate
const clicks = await ctx.db
  .query("clicks")
  .withIndex("by_affiliate",
    (q) => q.eq("affiliateId", args.affiliateId)
  )
  .order("desc")
  .take(20);

// Find most recent click within window
for (const click of clicks) {
  if (click._creationTime < cutoffTime) break;
  return { _id: click._id, referralLinkId: click.referralLinkId };
}
Set the attribution window — how far back to look for clicks. Defaults to 30 days but can be customized per campaign.
Fetch the 20 most recent clicks for this affiliate, ordered newest first. We don't need to scan the entire table — just recent ones.
Walk through clicks until we find one within the attribution window. If the click is too old, stop looking. Return the most recent valid click to link the conversion to.

Stage 3: The Commission — "Here's What You Earned"

Once a conversion is recorded, the system calculates how much the affiliate earns. This is where the campaign's commission rules come into play.

1

Check campaign status

Is the campaign still active? Paused and archived campaigns cannot earn new commissions. This prevents paying commissions for products that are no longer being promoted.

2

Calculate the commission amount

Based on the campaign's commission type: percentage (e.g., 20% of sale) or flat fee (e.g., $50 per sale). If the affiliate has a custom override rate, that takes priority.

3

Run self-referral detection

Check if the customer might be the affiliate themselves trying to earn a commission on their own purchase. If detected, the commission is flagged for review.

4

Set commission status

Commissions start as Pending by default. If the campaign has auto-approval enabled and the amount is below the threshold, it becomes Approved immediately.

// convex/commissions.ts
// createCommissionFromConversionInternal

// Get effective rate (override or campaign default)
const effectiveRate = affiliate
  .commissionOverrides?.find(
    (o) => o.campaignId === args.campaignId
        && o.status === "active"
  )?.rate ?? campaign.commissionValue;

// Calculate based on campaign type
let commissionAmount;
if (campaign.commissionType === "percentage") {
  commissionAmount = saleAmount * (effectiveRate / 100);
} else {
  // flatFee — the rate IS the amount
  commissionAmount = effectiveRate;
}
Find the commission rate — first check if this affiliate has a custom override for this campaign. If yes, use the override. If not, use the campaign's default rate.
Calculate the amount — if it's a percentage campaign, multiply the sale amount by the rate. If it's a flat fee, the rate IS the commission amount regardless of sale size.
💡
Commission Status Flow

Commissions follow a strict lifecycle: Pending → Approved → Paid. They can also go to Reversed (if a refund happens) or Declined (if the owner rejects them). Once a commission is paid, it's linked to a payout batch.

Stage 4: Commission Approval — "Is This Legitimate?"

Not every commission should be paid automatically. The system has multiple layers of review:

✅

Auto-Approval

If the campaign has autoApproveCommissions enabled and the commission is below the approvalThreshold, it's approved automatically. Small commissions don't need human review.

🔍

Manual Review

Large commissions (above the threshold) or those flagged for fraud require the SaaS Owner to manually review and approve them before they can be paid out.

🚫

Reversal

If a customer requests a refund or files a chargeback, the system automatically reverses the associated commission. The affiliate doesn't get paid for refunded sales.

convex/commissions.ts — approveCommission 
// Manual approval — SaaS Owner reviews a pending commission

// 1. Verify user is authenticated
const user = await getAuthenticatedUser(ctx);

// 2. Verify commission exists and belongs to this tenant
const commission = await ctx.db.get(args.commissionId);
if (commission.tenantId !== user.tenantId) {
  throw new Error("Unauthorized");
}

// 3. Only pending commissions can be approved
if (commission.status !== "pending") {
  throw new Error("Only pending commissions can be approved");
}

// 4. Update status to approved
await ctx.db.patch(args.commissionId, {
  status: "approved",
});

// 5. Log audit trail and send notification email

Stage 5: The Payout — "Getting Paid"

Once commissions are approved, they're grouped into payout batches and the affiliate receives their money. Here's how it works:

1

Generate a payout batch

The SaaS Owner clicks "Generate Payout." The system finds all approved commissions that haven't been paid yet and groups them by affiliate.

2

Create individual payout records

Each affiliate gets one payout record in the batch with their total amount. The system also creates a CSV file that can be used for bank transfers.

3

Mark payouts as paid

The owner processes the payments (via bank transfer, PayPal, etc.) and marks each payout as paid in the system. This can be done individually or for the entire batch at once.

4

Update commissions and notify

All commissions in the batch are updated to Paid status. The affiliate receives an email notification that their payment has been sent.

// convex/payouts.ts — generatePayoutBatch

// Find all approved, unpaid commissions
const commissions = await ctx.db
  .query("commissions")
  .withIndex("by_tenant_and_status",
    (q) => q.eq("tenantId", tenantId)
           .eq("status", "approved")
  )
  .collect();

// Filter out commissions already in a batch
const unpaidCommissions = commissions
  .filter((c) => !c.batchId);

// Group by affiliate and sum amounts
for (const comm of unpaidCommissions) {
  const existing = affiliateCommissions
    .get(comm.affiliateId) ?? { amount: 0, count: 0 };
  existing.amount += comm.amount;
  existing.count += 1;
  affiliateCommissions.set(comm.affiliateId, existing);
}
Find all approved commissions for this tenant — these are commissions that have been reviewed and are ready to be paid.
Filter out already-paid commissions — if a commission already has a batchId, it's been included in a previous payout. Don't pay twice!
Group by affiliate — for each affiliate, sum up all their unpaid commissions. This gives us the total amount each affiliate is owed.
💡
Tier Limits Apply to Payouts Too

The system enforces plan-based limits on how many payouts can be generated per month. A "Starter" plan might allow only 2 payout batches per month, while "Scale" allows unlimited. This prevents abuse and encourages plan upgrades.

🧠 Quick Check — Module 3

Q1: A customer clicks an affiliate link but doesn't buy anything for 45 days. The campaign's cookie duration is 30 days. What happens?

✅ Exactly! The cookie has an expiration window (30 days by default). After that window closes, the system can no longer attribute the sale to the affiliate. The conversion is still recorded as "organic" for analytics, but no commission is generated.
Not quite. Cookie duration is a hard limit — once it expires, the attribution link is broken. The system doesn't extend cookies automatically. This is actually configurable per campaign, but the default is 30 days and it's strictly enforced.

Q2: You notice a commission is stuck in "pending" status. What are the possible reasons?

✅ Right! Commissions stay pending when: (1) auto-approval is disabled on the campaign, (2) the amount exceeds the threshold requiring manual review, or (3) fraud detection flagged it. The SaaS Owner needs to manually approve it before it can be paid.
Think about the commission approval flow. Auto-approval isn't always enabled, and large commissions often require human review. Also, fraud detection can flag commissions for manual review. The payout method doesn't affect commission status — it only affects whether the payout can be processed.
Module 4

Campaign Types & Commission Structures

Not all affiliate programs pay the same way. Campaigns define the rules of the game — how much affiliates earn, how long cookies last, and whether recurring payments generate ongoing commissions.

Two Ways to Pay Affiliates

Every campaign uses one of two commission types. This is the most fundamental decision a SaaS Owner makes when setting up an affiliate program.

📊 Percentage Commission

The affiliate earns a percentage of each sale. If the sale is ₱5,000 and the rate is 20%, the affiliate earns ₱1,000.

Best for: Products with varying prices. The affiliate's earnings scale with the sale amount.

💵 Flat Fee Commission

The affiliate earns a fixed amount per sale, regardless of the sale price. Every conversion earns exactly ₱500.

Best for: Products with a single price point or when you want predictable commission costs.

convex/commissions.ts — Commission calculation 
// The actual calculation logic

if (campaign.commissionType === "percentage") {
  // ₱5,000 sale × 20% = ₱1,000 commission
  commissionAmount = saleAmount * (effectiveRate / 100);
} else {
  // Flat fee — always ₱500 regardless of sale amount
  commissionAmount = effectiveRate;
}

Recurring Commissions — "Earn on Every Renewal"

For subscription products, affiliates can earn commissions not just on the initial sale, but on every renewal too. This is a powerful incentive for affiliates to promote subscription products.

💡 Think of it like this

Imagine a real estate agent who doesn't just get paid when you buy a house — they also get a small cut every month you live in it. That's recurring commissions. The affiliate keeps earning as long as the customer stays subscribed.

Campaigns can configure recurring commissions in three ways:

Rate Type How It Works Example
Same Recurring commissions use the same rate as the initial sale 20% on signup, 20% on every renewal
Reduced Recurring commissions use a lower rate (default 50% of initial) 20% on signup, 10% on every renewal
Custom Recurring commissions use a specific rate set by the owner 20% on signup, 15% on every renewal 
// convex/commissionEngine.ts
// calculateRecurringCommissionAmount

switch (campaign.recurringRateType) {
  case "same":
    rate = campaign.commissionValue;
    break;
  case "reduced":
    // Default: 50% of initial rate
    rate = campaign.recurringRate
      ?? (campaign.commissionValue * 0.5);
    break;
  case "custom":
    rate = campaign.recurringRate
      ?? campaign.commissionValue;
    break;
}

if (campaign.commissionType === "percentage") {
  return saleAmount * (rate / 100);
} else {
  return rate;
}
Determine the recurring rate based on the campaign's configuration. "Same" uses the original rate, "reduced" halves it (or uses a custom reduced rate), and "custom" uses whatever rate the owner specified.
Apply the rate using the same percentage or flat-fee logic as the initial commission. The calculation method is consistent — only the rate changes.
📋
Subscription Lifecycle Events

The system handles four subscription events via webhooks: subscription.created (initial sale), subscription.updated (renewal/upgrade/downgrade), subscription.cancelled (customer left), and refund.created (money returned). Each triggers different commission logic.

Auto-Approval Thresholds — "Trust but Verify"

SaaS Owners can configure campaigns to automatically approve small commissions while requiring manual review for larger ones. This balances speed with control.

1

Auto-approval is enabled

The campaign has autoApproveCommissions: true.

2

Check the threshold

If a threshold is set (e.g., ₱1,000), commissions below that amount are auto-approved. Commissions at or above the threshold require manual review.

3

No threshold = approve all

If auto-approval is enabled but no threshold is set, ALL commissions are auto-approved regardless of amount.

⚠️
Fraud Detection Overrides Auto-Approval

Even if a commission would normally be auto-approved, if the self-referral detection system flags it, the commission is forced into "pending" status for manual review. Fraud detection always takes priority.

Per-Affiliate Commission Overrides

Sometimes you want to give a specific affiliate a different commission rate than the campaign default. Maybe they're a top performer who negotiated a better deal, or maybe they're new and you want to offer a trial rate.

convex/schema.ts — commissionOverrides field

✅ Right! The "reduced" rate type defaults to 50% of the initial rate. So 20% → 10% for renewals. 10% of ₱1,000 = ₱100. The affiliate earns ₱200 on the first month and ₱100 on each subsequent month.

Module 5

Fraud Detection — Keeping the System Honest

When money is involved, some people will try to game the system. This module explains how the platform detects and prevents affiliate fraud — specifically the most common type: self-referral.

What Is Self-Referral Fraud?

Self-referral fraud happens when an affiliate tries to earn a commission on their own purchase. Imagine being a salesperson who gets a commission every time you buy something from your own store — that's exactly what this is.

💡 Think of it like this

It's like a restaurant waiter who rings up their own meals as "customer orders" so they can collect tips from themselves. The restaurant is paying out money for sales that didn't actually come from real customers.

The system detects self-referral by comparing data points between the affiliate and the customer:

📧

Email Match (3 points)

The customer's email matches the affiliate's email. This is the strongest signal — if the buyer and the affiliate share an email, it's almost certainly self-referral.

🌐

IP Match (3 points) or Subnet Match (1 point)

The customer's IP address matches the affiliate's login IP. A subnet match (same network, like same office or home WiFi) is a weaker signal.

💻

Device Fingerprint Match (2 points)

The customer's device fingerprint matches the affiliate's device. This catches people using the same computer for both accounts.

💳

Payment Method Match (2 points)

The customer's payment method (last 4 digits or processor ID) matches the affiliate's payout method. Same credit card = same person.

💡
Weighted Scoring System

The system uses a weighted scoring approach. Each matched indicator contributes points. If the total score reaches 3 or higher, the conversion is flagged as self-referral. This is calibrated for the Philippine/SEA market where shared networks (office WiFi, family devices) are common — a single subnet match alone (1 point) won't trigger a false positive.

How Detection Works in the Code

Self-referral detection runs automatically every time a commission is created. It's built into the createCommissionFromConversionInternal function.

// convex/fraudDetection.ts — detectSelfReferral

// Compare email (case-insensitive)
if (conversion.customerEmail
    && affiliate.email) {
  if (conversion.customerEmail
      .toLowerCase()
      === affiliate.email.toLowerCase()) {
    matchedIndicators.push("email_match");
  }
}

// Compare IP address (exact or /24 subnet)
if (conversion.ipAddress
    && affiliate.lastLoginIp) {
  if (conversion.ipAddress
      === affiliate.lastLoginIp) {
    matchedIndicators.push("ip_match");
  } else if (isSameSubnet(
    conversion.ipAddress,
    affiliate.lastLoginIp, 24
  )) {
    matchedIndicators.push("ip_subnet_match");
  }
}

// Calculate weighted score
const totalScore = matchedIndicators
  .reduce((sum, indicator) =>
    sum + (SIGNAL_WEIGHTS[indicator] ?? 1), 0
  );

const isSelfReferral = totalScore >= 3;
Check email match — compare the customer's email with the affiliate's email, ignoring case. If they match, add "email_match" (3 points).
Check IP match — first check for an exact IP match (3 points). If not exact, check if they're on the same /24 subnet (1 point). This catches shared networks without false-flagging entire offices.
Calculate the score — add up the weights of all matched indicators. Email match alone (3 points) triggers detection. IP subnet + device match (1+2=3) also triggers it.

What Happens When Fraud Is Detected

When self-referral is detected, the system takes several actions automatically:

1

Commission is flagged

The commission gets isSelfReferral: true and fraudIndicators listing which signals matched (e.g., ["email_match", "ip_match"]).

2

Commission stays pending

Even if the campaign has auto-approval enabled, self-referral commissions are forced into "pending" status for manual review.

3

Fraud signal is added to affiliate record

A fraud signal entry is added to the affiliate's fraudSignals array with details about what was detected and when.

4

SaaS Owner is notified

A fraud alert email is scheduled to be sent to the SaaS Owner, informing them of the suspicious activity.

5

Audit trail is logged

Commission Reversal — "Taking Back the Money"

Sometimes a commission needs to be reversed after it's been created. This happens in two scenarios:

↩️

Refund

The customer requested and received a refund. The commission for that sale must be reversed since the sale no longer exists.

🏦

Chargeback

The customer disputed the charge with their bank. The payment was forcibly reversed, and the commission must be reversed too. Chargebacks also add a fraud signal.

🚨
Only Reversible Commissions

Only commissions with status Pending or Approved can be reversed. Once a commission is Paid, it's already been sent to the affiliate and can't be automatically reversed — that would require a separate recovery process.

Key Convex Patterns Used Throughout

Understanding these patterns will help you read and modify any file in this codebase:

🔍

Queries (read data)

Use query to read data from the database. They're reactive — when data changes, the UI updates automatically. Always use indexes for efficient reads.

✏️

Mutations (write data)

Use mutation to create, update, or delete data. They run in transactions — either everything succeeds or everything fails. No partial updates.

🌐

Actions (external calls)

Use action to call external APIs (like sending emails or processing webhooks). They run in Node.js, not in the database runtime, so they can use any npm package.

🔒

Internal Functions

Use internalQuery, internalMutation, and internalAction for functions that should only be called by other Convex functions, not directly from the frontend.

The standard function pattern used everywhere 
// Every Convex function follows this pattern:

export const myFunction = query({
  // 1. Define what arguments it accepts
  args: {
    affiliateId: v.id("affiliates"),
  },
  // 2. Define what it returns
  returns: v.object({
    name: v.string(),
    email: v.string(),
  }),
  // 3. The actual logic
  handler: async (ctx, args) => {
    const affiliate = await ctx.db.get(args.affiliateId);
    return { name: affiliate.name, email: affiliate.email };
  },
});

Q1: You need to add a new field to track an affiliate's social media profile URL. Where do you make this change?

Q2: Why are webhook processing functions implemented as "actions" rather than "mutations"?

✅ Right! Webhook processing needs to make HTTP calls (to normalize payloads, call external services, etc.). Actions run in the Node.js runtime which supports the full Node.js ecosystem. Mutations run in Convex's V8 runtime which doesn't support HTTP calls or dynamic imports.
The key difference is the runtime environment. Actions run in Node.js (full ecosystem access), while mutations run in Convex's V8 isolate (database-only, no external calls). Webhook processing needs to handle HTTP requests and potentially call external services.
